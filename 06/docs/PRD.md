## 1. 제품 개요

### 1.1 한 줄 정의

마크다운 문서 한 장을 입력하면 한국어 9:16 숏폼 영상(MP4)을 자동 생성하는 로컬 CLI 제작 도구.

### 1.2 배경

뉴스레터, 리서치 메모, 블로그 글, 스토리보드 초안은 대부분 텍스트나 마크다운으로 축적된다. 하지만 이를 YouTube Shorts, Reels, TikTok 같은 숏폼 채널로 재배포하려면 대본 작성, 녹음, 편집, 자막, 검수, 렌더링을 매번 수동 반복해야 한다. MD2Short는 이 반복 작업을 로컬 job 단위로 자동화한다.

### 1.3 v1 비전

- 마크다운 1개 → 업로드 가능한 한국어 숏폼 영상 1편을 생성한다.
- 실제 TTS 오디오와 Scribe 단어 타임스탬프를 기준으로 자막·시각 cue를 동기화한다.
- 호랑이 캐릭터, 숫자 카드, 오프닝, 회사 로고 cue, 선택적 photo mode로 반복 가능한 브랜드 포맷을 만든다.
- 모든 산출물을 `jobs/<job_id>/` 아래에 남겨 재현·디버깅·비용 추적이 가능하게 한다.

### 1.4 차별점

- **마크다운 친화적 입력**: 일반 본문 또는 `### N컷` 스토리보드 형식을 별도 CMS 없이 처리한다.
- **음성 기준 자막 sync**: `words.json` 단어 타이밍을 단일 진실원으로 사용한다.
- **브랜드형 시각 시스템**: 호랑이 PNG cue, 숫자 강조 카드, 회사 로고, 오프닝/CTA를 고정 포맷으로 조합한다.
- **로컬 우선 제어권**: 원본 문서, API 로그, 중간 산출물, 최종 MP4가 사용자의 로컬 작업 공간에 남는다.
- **안전한 확장 경계**: photo mode 실패 시 emoticon mode로 fallback하고, SaaS/API/다국어는 v2로 분리한다.

## 2. 문제 정의

| 문제 | 현상 | v1 대응 |
| --- | --- | --- |
| 글 자산이 영상으로 전환되지 못함 | 짧은 브리핑도 녹음·편집·자막에 시간이 큼 | CLI 한 번으로 대본·TTS·자막·렌더링을 수행 |
| 자막·음성 sync 어긋남 | 추정 타이밍이나 수동 편집에 의존 | Scribe `words.json`을 단일 시간 기준으로 사용 |
| 브랜드 일관성 유지 어려움 | 영상마다 다른 편집 스타일 적용 | Remotion 고정 레이아웃 + 반복 가능한 cue 시스템 제공 |
| 경제·리서치 수치 전달 어려움 | 숫자·지표가 자막 속에 묻힘 | 숫자 카드와 원문 숫자 표시 복원으로 강조 |
| 제작 품질 판단 부담 | 업로드 전 확인 항목이 많음 | QC 리포트, 상태 전이, 로그를 job별로 저장 |

## 3. 목표와 비목표

### 3.1 목표 (Goals, v1)

| ID | 목표 |
| --- | --- |
| **G1** | 한국어 마크다운 또는 스토리보드 입력을 9:16 MP4로 변환한다. |
| **G2** | 자막·시각 cue timing은 실제 TTS 오디오에서 추출한 `words.json`을 기준으로 한다. |
| **G3** | 목표 초수 입력 없이 입력 분량과 실제 TTS 길이에 맞는 영상 길이를 사용한다. (정보 보존 우선; 길이 강제 금지) |
| **G4** | v1 출력 언어는 한국어로 고정하고, en/ja는 v2 roadmap으로 둔다. |
| **G5** | 자동 QC가 오디오 tail, 렌더 길이, 비용, 산출물 존재 여부를 검증한다. |
| **G6** | 모든 중간 산출물과 로그를 `jobs/<job_id>/`에 남겨 재현 가능한 로컬 워크플로를 제공한다. |
| **G7** | 비전문 편집자도 README와 `.env` 설정만으로 첫 영상을 생성할 수 있게 한다. |

### 3.2 비목표 (Non-Goals, v1)

- 한국어 외 다국어 출력(en/ja 등)은 v1 범위가 아니다.
- 한 원문에서 여러 후보를 자동 생성하는 batch/variant 기능은 v1 범위가 아니다.
- 기존 job의 자막·CTA만 부분 수정해 중간 산출물을 재사용하는 부분 재렌더링은 v1 범위가 아니다.
- HTTP API, 다중 사용자 계정, SaaS 운영, 클라우드 큐·분산 렌더링은 v1 범위가 아니다.
- 인터뷰·강의 원본 영상의 컷 편집, 실시간 스트리밍 생성, 딥페이크·아바타 영상 생성은 범위가 아니다.
- 음악 작곡·BGM 생성은 범위가 아니다. 필요한 경우 라이선스가 확보된 BGM만 별도 설계한다.
- SNS 자동 업로드는 v2 이후 범위다.

## 4. 타겟 사용자

### 4.1 Primary Persona — 한국어 경제·리서치 브리핑 운영자

- 매일 리서치 메모나 뉴스 요약을 마크다운으로 정리한다.
- 콘텐츠 기획력은 있지만 영상 편집 시간과 음성 녹음 시간이 병목이다.
- 로컬 CLI, `.env`, API key 설정을 사용할 수 있다.
- 일정한 브랜드 포맷으로 숏폼을 반복 발행하고 싶다.

### 4.2 Secondary Persona — 기술 콘텐츠 크리에이터 / 1인 미디어 운영자

- 블로그, 뉴스레터, 강의 노트를 마크다운으로 작성한다.
- 전문 편집자 없이도 자막·오프닝·숫자 강조가 있는 짧은 영상을 만들고 싶다.
- 향후에는 같은 원문에서 여러 후킹·템플릿 변형을 만들 필요가 있다.

### 4.3 Tertiary Persona — 기업 마케팅·에이전시 (v2 이후)

- 다수 클라이언트 또는 캠페인의 텍스트 콘텐츠를 대량 영상화해야 한다.
- 팀 협업, API 연동, 권한 관리, 자동 업로드가 필요하다.
- v1에서는 직접 타겟하지 않고 v2 확장 가능성으로만 고려한다.

## 5. 사용자 시나리오

### 5.1 시나리오 1 — 리서치 메모 숏폼화

1. 사용자가 당일 리서치 메모를 `sample.md` 같은 일반 마크다운으로 준비한다.
2. 시스템이 한국어 브리핑 대본을 만들거나 fallback extract로 원문을 보존한다.
3. 시스템이 TTS, Scribe, cue planning, Remotion render를 수행한다.
4. 사용자는 `jobs/<job_id>/edit/final.mp4`, `output/ret.mp4`, `qc_report.json`을 확인한다.

### 5.2 시나리오 2 — 스토리보드 기반 컷 지정

1. 사용자가 `### 1컷` 형식으로 `자막`, `화면 연출`, `이미지 프롬프트`를 작성한다.
2. 시스템은 `자막`만 내레이션에 사용하고, 화면 연출/프롬프트는 시각 cue 선택에만 사용한다.
3. 각 컷은 실제 `words.json` 문장 boundary에 맞춰 Remotion `captionSegments`로 고정된다.

### 5.3 시나리오 3 — 설정 변경 후 새 job 생성

1. 사용자가 배경색, voice, speed, visual mode, output path 같은 CLI 옵션을 바꾼다.
2. 시스템은 새 `job_id`로 전체 파이프라인을 다시 실행한다.
3. 사용자는 이전 job과 새 job의 로그/QC/최종 영상을 비교한다.

### 5.4 시나리오 4 — photo mode 실사 이미지 생성

1. 사용자가 `--visual-mode photo`와 `OPENAI_API_KEY`를 제공한다.
2. 시스템은 자막 segment별 영어 이미지 prompt를 만들고 `gpt-image-2`로 9:16 이미지를 생성한다.
3. 이미지 생성이 moderation 또는 API 오류로 실패하면 emoticon mode로 자동 fallback한다.

## 6. 기능 요구사항

| ID | 기능 | 사용자 가치 | 제품 요구 |
| --- | --- | --- | --- |
| **F-01** | 마크다운·스토리보드 입력 처리 | 원문을 별도 편집툴 없이 사용 | 일반 markdown과 `### N컷` 스토리보드 입력을 구분하고 원본을 `inputs/source.md`에 저장한다. |
| **F-02** | 한국어 대본 생성/추출 | 별도 카피라이팅 없이 내레이션 확보 | OpenAI Responses API 또는 fallback extract로 한국어 대본을 만들고, v1에서 외국어 문장 leak은 거부한다. |
| **F-03** | 한국어 TTS 내레이션 | 녹음 없이 음성 확보 | ElevenLabs 한국어 voice로 단일 `audio/narration.mp3`를 생성하고, TTS 전용 텍스트는 숫자·약어 발음과 pause-heavy punctuation을 보정한다. |
| **F-04** | 단어 단위 타임스탬프 | 자막 sync 기준 확보 | ElevenLabs Scribe 결과를 ms 단위 `transcripts/words.json`으로 저장하고 모든 timing의 단일 진실원으로 사용한다. |
| **F-05** | 자동 편집 cue planning | 수동 컷 편집 부담 감소 | 자막 segment, 호랑이/photo cue, 숫자 카드, 회사 로고 cue를 `words.json` 기반으로 생성한다. |
| **F-06** | 브랜드형 9:16 렌더링 | 업로드 가능한 영상 확보 | Remotion 단일 렌더 경로로 자막 포함 MP4를 만들고, ffmpeg는 오디오 길이 측정·QC 같은 보조 도구로만 사용한다. |
| **F-07** | 자동 QC·상태·로그 | 오류와 비용 추적 | `status.json`, `qc_report.json`, 단계별 logs를 job별로 저장하고 QC 실패를 terminal 상태로 기록한다. |
| **F-08** | 실사 photo mode | 브리핑별 이미지 다양성 | `--visual-mode photo`에서 자막 segment의 핵심 키워드/주체를 중심으로 9:16 PNG를 생성한다. 예: `트럼프가 합의를 했다`는 Donald Trump를 primary subject로 삼는다. 외부 이미지 API가 차단하면 sanitised/fallback prompt 후 emoticon mode로 fallback한다. |
| **F-09** | 호랑이 캐릭터 cue | 무료 deterministic 브랜드 시각 | 키워드/문장 span 기반으로 로컬 호랑이 PNG를 배치한다. |
| **F-10** | 숫자·지표 강조 카드 | 경제·리서치 수치 전달 | 퍼센트, 금액, 지수, 포인트 같은 수치를 추출해 NumberCard로 표시한다. |
| **F-11** | Opening / Outro | 반복 포맷과 CTA | 날짜 기반 오프닝을 기본 표시하고, Outro CTA는 명시 옵션일 때만 표시한다. |
| **F-12** | 결과물 경로와 job 디버깅 | 산출물 추적 | `jobs/<job_id>/edit/final.mp4`, 기본 `output/ret.mp4`, logs, staging props를 일관된 위치에 남긴다. |

### 6.1 v2 후보 기능

- en/ja 등 다국어 대본·voice·Scribe 언어 확장.
- 한 원문에서 여러 hook/template 후보를 생성하는 batch/variant 기능.
- 기존 job의 일부 산출물을 재사용하는 부분 수정·재렌더링.
- HTTP API, 큐, 클라우드 렌더링, 팀 협업, SNS 자동 업로드.

## 7. 비기능 요구사항

| 항목 | 요구사항 | 검증 기준 |
| --- | --- | --- |
| 성능 | 로컬 단일 job이 사용자가 기다릴 수 있는 시간 안에 종료되어야 한다. 5분 p95는 베타 목표이며 별도 벤치마크로 검증한다. | 샘플 세트 처리 시간 측정 |
| 처리량 | 1인 운영자가 반복 실행 가능한 job 기반 흐름이어야 한다. 하루 30편은 운영 목표이며 현재 보장값이 아니다. | 반복 생성 시나리오 측정 |
| 자막 정확도 | 자동 QC는 last word vs audio tail ±1000ms, final MP4 vs audio ±100ms를 기준으로 한다. ±50ms sync는 장기/수동 검수 목표로 분리한다. | `qc_report.json` + 샘플 수동 검수 |
| 비용 | 기본 emoticon mode는 30초 영상 1편당 외부 API 비용 목표 $3.00 이하를 유지한다. photo mode 이미지 생성 비용은 로컬 추정에 합산하지 않고 실제 API 청구서 기준으로 확인한다. | job 단위 텍스트·음성 비용 추정 및 청구서 대조 |
| 보안 | API key와 원본 마크다운은 사용자가 통제하는 로컬 환경에서 관리되어야 한다. | `.env`, git 제외, 로그 노출 검수 |
| 재현성 | 동일 job의 중간 산출물과 설정을 추적할 수 있어야 한다. | `jobs/<job_id>/` 산출물 확인 |
| 관측 가능성 | 실패 원인, 비용, 처리 단계, QC 결과를 사용자가 확인할 수 있어야 한다. | `status.json`, `qc_report.json`, `logs/` 확인 |
| 사용성 | 비전문 편집자도 README만 보고 첫 영상을 만들 수 있어야 한다. | 신규 사용자 설치·실행 테스트 |
| 라이선스 | 사용 폰트, Remotion, 이미지/BGM 등 배포에 영향을 주는 라이선스가 명확해야 한다. | 베타 전 라이선스 체크리스트 |

> 동시 100건, 1일 1,000건, 다중 사용자 권한 관리 등 SaaS 운영 요구는 v2에서 별도 PRD로 다룬다.

## 9. 미해결 질문

제품 의사결정 단계에서 결론낼 항목.

- BGM은 자체 라이브러리를 구축할지, 외부 라이선스 음원을 사용할지.
- v1 로컬 도구에서 v2 SaaS/API로 넘어가는 기준은 무엇인지.
